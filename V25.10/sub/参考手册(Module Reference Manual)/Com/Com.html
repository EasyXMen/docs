з®Җд»Ӣ(Introduction)В¶

ComжЁЎеқ—дё»иҰҒе®һзҺ°дәҶSignalеңЁI-PDUдёӯзҡ„е°ҒиЈ…еҸҠи§ЈжһҗеҠҹиғҪпјҢдёәRTEеұӮжҸҗдҫӣдәҶеҹәдәҺSignalзҡ„еҸ‘йҖҒдёҺжҺҘж”¶жҺҘеҸЈпјҢе®һзҺ°дәҶеҹәдәҺSignalзҡ„зҪ‘е…іеҠҹиғҪпјҢе®һзҺ°дәҶPDUзҡ„дёҚеҗҢеҸ‘йҖҒжЁЎејҸпјҢд»ҘеҸҠSignalж»ӨжіўпјҢUpdate bitзӯүеҠҹиғҪгҖӮ

The Com module mainly implements the functions of encapsulating and parsing Signals in I-PDUs, provides Signal-based sending and receiving interfaces for the RTE layer, realizes Signal-based gateway functions, implements different sending modes of PDUs, as well as functions such as Signal filtering and Update bit.

еҰӮеӣҫ :[Com_fig_arch]жүҖзӨәпјҢComжЁЎеқ—еӨ„дәҺAUTOSARжһ¶жһ„дёӯзҡ„йҖҡдҝЎжңҚеҠЎеұӮпјҢе…¶дёӢеұӮжЁЎеқ—дёәPduRжЁЎеқ—пјҢдёҠеұӮжЁЎеқ—дёәRTEгҖӮ

As shown in the [Module Hierarchy Diagram], the Com module is located in the Communication Service Layer of the AUTOSAR architecture, with the PduR module as its lower-layer module and the RTE as its upper-layer module.

I-PDU(Group)В¶

ComжЁЎеқ—е®һзҺ°еҹәдәҺI-PDU Groupзҡ„дҪҝиғҪжҺ§еҲ¶пјҢжҺҘж”¶и¶…ж—¶жЈҖжөӢдҪҝиғҪжҺ§еҲ¶гҖӮж №жҚ®I-PDUдёҺI-PDU Groupзҡ„еҢ…еҗ«е…ізі»пјҢй—ҙжҺҘе®һзҺ°еҜ№еҗ„дёӘI-PDUзҡ„йҖҡдҝЎдҪҝиғҪжҺ§еҲ¶еҸҠRx I-PDUзҡ„жҺҘж”¶и¶…ж—¶жЈҖжөӢдҪҝиғҪжҺ§еҲ¶гҖӮ

The Com module implements enable control based on I-PDU Group and enable control for reception timeout detection. According to the inclusion relationship between I-PDUs and I-PDU Groups, it indirectly realizes the communication enable control for each I-PDU and the enable control for reception timeout detection of Rx I-PDUs.

Signal е°ҒиЈ…и§Јжһҗ(Signal encapsulation and parsing)В¶

  • дҝЎеҸ·зҡ„е°ҒиЈ…е’Ңи§ЈжһҗжҳҜComжЁЎеқ—зҡ„ж ёеҝғеҠҹиғҪпјҢж №жҚ®еҗ„дёӘдҝЎеҸ·зҡ„й…ҚзҪ®дҝЎжҒҜе°ҶеҸ‘йҖҒSignalе°ҒиЈ…еҲ°е…іиҒ”зҡ„Tx IPduж•°жҚ®дёӯпјҢд»ҺRx IPduж•°жҚ®дёӯи§ЈжһҗжҺҘж”¶SignalгҖӮеҢ…жӢ¬еҜ№йқһеҠЁжҖҒй•ҝеәҰзұ»еһӢгҖҒеҠЁжҖҒй•ҝеәҰзұ»еһӢзҡ„SignalгҖҒGroupSignalгҖҒеӯ—иҠӮеҜ№йҪҗзҡ„SignalGroupзҡ„ж”¶еҸ‘гҖӮ

    The encapsulation and parsing of signals are the core functions of the Com module. It encapsulates the transmitted signals into the associated Tx IPdu data according to the configuration information of each signal, and parses the received signals from the Rx IPdu data. This includes the transmission and reception of non-dynamic length type signals, dynamic length type signals, GroupSignals, and byte-aligned SignalGroups.

  • ComжЁЎеқ—дёәRTE/еә”з”ЁеұӮжҸҗдҫӣдәҶе®Ңж•ҙзҡ„еҹәдәҺSignal/SignalGroupзҡ„ж”¶еҸ‘жҺҘеҸЈгҖӮ

    The Com module provides the RTE/application layer with complete sending and receiving interfaces based on Signal/SignalGroup.

IPdu ж”¶еҸ‘(IPdu transmission and reception)В¶

  • ComжЁЎеқ—е®һзҺ°IPduзҡ„ж”¶еҸ‘ж–№ејҸжҢүж•°жҚ®жөҒеҲҶдёӨз§Қзұ»еһӢпјҢеҚіIFж–№ејҸе’ҢTPж–№ејҸпјҢIFж–№ејҸйҖҡеёёз”ЁдәҺвҖңж•°жҚ®й•ҝеәҰиҫғе°ҸвҖқзҡ„IPduпјҢиҖҢTPж–№ејҸйҖҡеёёз”ЁдәҺвҖңж•°жҚ®й•ҝеәҰиҫғеӨ§вҖқзҡ„IPduпјҢиҝҷйҮҢзҡ„вҖңж•°жҚ®й•ҝеәҰвҖқжҳҜзӣёеҜ№дәҺдј иҫ“жҖ»зәҝжқҘе®ҡд№үзҡ„пјҢеҰӮCANжҖ»зәҝдёә8еӯ—иҠӮпјҢCANFDдёә64еӯ—иҠӮпјҢETHеҸҜд»ҘиҫҫеҲ°1000+еӯ—иҠӮгҖӮе…¶дёӯIF IPduзҡ„еҸ‘йҖҒеҸҲеҲҶдёәDirectе’ҢTriggerTransmitдёӨз§Қзұ»еһӢпјҢеүҚиҖ…еҸ‘йҖҒж—¶жңәз”ұComеҶіе®ҡпјҢеҗҺиҖ…еҸ‘йҖҒж—¶жңәз”ұдёӢеұӮжЁЎеқ—еҶіе®ҡгҖӮ

    The Com module implements two types of IPdu transmission and reception methods based on data flow: the IF mode and the TP mode. The IF mode is usually used for IPdus with вҖңsmall data lengthвҖқ, while the TP mode is generally applied to IPdus with вҖңlarge data lengthвҖқ. Here, the вҖңdata lengthвҖқ is defined relative to the transmission bus. For example, the CAN bus is 8 bytes, CANFD is 64 bytes, and ETH can reach more than 1000 bytes. Among them, the transmission of IF IPdu is further divided into two types: Direct and TriggerTransmit. The transmission timing of the former is determined by Com, while that of the latter is decided by the lower-layer module.

  • Tx IPduд»ҺеҸ‘йҖҒж—¶жңәи§’еәҰеҸҲеҲҶдёәеӣӣз§ҚжЁЎејҸпјҢеҚіPERIODICпјҢDIRECTпјҢMIXEDпјҢNONEгҖӮNONEжЁЎејҸйҖҡеёёдёҺTriggerTransmitпјҢжҲ–иҖ…и°ғз”ЁCom_TriggerIPDUSend/ Com_TriggerIPDUSendWithMetaDataжқҘй…ҚзҪ®е®һзҺ°IPduзҡ„еҸ‘йҖҒгҖӮ

    From the perspective of transmission timing, Tx IPdu is further divided into four modes: PERIODIC, DIRECT, MIXED, and NONE. The NONE mode is usually configured with TriggerTransmit, or by calling Com_TriggerIPDUSend/Com_TriggerIPDUSendWithMetaData to implement the transmission of IPdu.

и¶…ж—¶зӣ‘жөӢеҠҹиғҪ(Timeout monitoring function)В¶

  • ComжЁЎеқ—е®һзҺ°еҹәдәҺSignal/SignalGroupзҡ„и¶…ж—¶зӣ‘жөӢеҠҹиғҪгҖӮеҜ№дәҺеҸ‘йҖҒе®һзҺ°и¶…ж—¶йҖҡзҹҘжңәеҲ¶пјҢеҜ№дәҺжҺҘж”¶е®һзҺ°и¶…ж—¶йҖҡзҹҘе’Ңи¶…ж—¶жӣҝжҚўдёӨз§ҚжңәеҲ¶гҖӮ

    The Com module implements a timeout monitoring function based on Signal/SignalGroup. For transmission, it implements a timeout notification mechanism; for reception, it implements two mechanisms: timeout notification and timeout replacement.

  • еҸ‘йҖҒи¶…ж—¶зӣ‘жөӢпјҡзӣ‘жөӢиҜ·жұӮеҸ‘йҖҒеҲ°е…¶жүҖеңЁзҡ„IPduеҸ‘йҖҒжҲҗеҠҹиҝҷж®өж—¶й—ҙпјҢIPduи¶…ж—¶ж—¶й—ҙйҳҲеҖјеҸ–иҜ·жұӮеҸ‘йҖҒзҡ„Signal/SignalGroupй…ҚзҪ®зҡ„и¶…ж—¶ж—¶й—ҙжңҖе°ҸеҖјгҖӮеҪ“еҸ‘з”ҹеҸ‘йҖҒи¶…ж—¶пјҢйҖҡзҹҘдёҠеұӮжЁЎеқ—гҖӮ

    Transmission timeout monitoring: Monitors the period from when a transmission request is made to when the transmission of the IPdu it belongs to is successful. The IPdu timeout threshold is the minimum timeout value configured for the Signal/SignalGroup that requested the transmission. When a transmission timeout occurs, the upper-layer module is notified.

  • жҺҘж”¶и¶…ж—¶зӣ‘жөӢпјҡзӣ‘жөӢдёӨж¬ЎжӯЈзЎ®жҺҘж”¶дҝЎеҸ·еҖјд№Ӣй—ҙзҡ„ж—¶й—ҙж®өпјҢSignal/SignalGroupи¶…ж—¶ж—¶й—ҙйҳҲеҖјж №жҚ®е…¶еҗ„иҮӘзҡ„й…ҚзҪ®еҸӮж•°еҶіе®ҡгҖӮеҪ“еҸ‘з”ҹи¶…ж—¶ж—¶пјҢиҝӣиЎҢйҖҡзҹҘжҲ–иҖ…е°ҶжҺҘж”¶дҝЎеҸ·еҖјжӣҝжҚўдёәеҲқе§ӢеҖјгҖҒжӣҝд»ЈеҖјгҖӮ

    Reception timeout monitoring: Monitors the time period between two successful receptions of signal values. The timeout threshold for a Signal/SignalGroup is determined according to its respective configuration parameters. When a timeout occurs, a notification is sent, or the received signal value is replaced with an initial value or a substitute value.

дҝЎеҸ·ж»ӨжіўеҠҹиғҪ(Signal filtering function)В¶

  • ComжЁЎеқ—е®һзҺ°еҹәдәҺдҝЎеҸ·зҡ„дҝЎеҸ·еҖјиҝӣиЎҢиҝҮж»Өзҡ„жңәеҲ¶пјҢиҷҪз„¶еҜ№дәҺж”¶еҸ‘дҝЎеҸ·зҡ„иҝҮж»Өз®—жі•дёҖж ·пјҢдҪҶиҝҮж»Өзҡ„зӣ®зҡ„е®Ңе…ЁдёҚеҗҢгҖӮ

    The Com module implements a mechanism for filtering based on signal values. Although the filtering algorithm for transmitting and receiving signals is the same, the purposes of filtering are completely different.

  • еҜ№дәҺжҺҘж”¶Signal/SignalGroupпјҢиҝҮж»Өз®—жі•жңӘйҖҡиҝҮж—¶пјҢе°ҶиҲҚејғеҪ“еүҚжҺҘж”¶еҲ°зҡ„Signal/SignalGroupгҖӮ

    For received Signals/SignalGroups, if the filtering algorithm fails, the currently received Signal/SignalGroup will be discarded.

  • еҜ№дәҺеҸ‘йҖҒSignal/GroupSignalпјҢиҝҮж»Өз®—жі•зҡ„з»“жһңеҸӘеҶіе®ҡиҜҘеҸ‘йҖҒIPduйҖүжӢ©ComTxModeTrueжҲ–иҖ…ComTxModeFalseиҝӣиЎҢжҠҘж–ҮеҸ‘йҖҒпјҢдёҚдјҡиҲҚејғдҝЎеҸ·жң¬иә«гҖӮ

    For transmitted Signals/GroupSignals, the result of the filtering algorithm only determines whether the transmitting IPdu selects ComTxModeTrue or ComTxModeFalse for message transmission, and the signal itself is not discarded.

дҝЎеҸ·UpdateеҠҹиғҪ(Signal Update function)В¶

  • ComжЁЎеқ—е®һзҺ°еҹәдәҺSignal/SignalGroupзҡ„Update BitжңәеҲ¶жқҘиҜҶеҲ«дҝЎеҸ·жҳҜеҗҰжӣҙж–°(дҝЎеҸ·жӣҙж–°пјҢдҪҶдҝЎеҸ·еҖјдёҚдёҖе®ҡеҸҳеҢ–)гҖӮ

    The Com module implements an Update Bit mechanism based on Signal/SignalGroup to identify whether a signal has been updated (signal update does not necessarily imply signal value change).

  • Signal/SignalGroupйҖҡиҝҮй…ҚзҪ®еҸӮж•°ComUpdateBitPositionжқҘеҶіе®ҡжҳҜеҗҰж”ҜжҢҒUpdateжңәеҲ¶пјҢUpdate Bitжң¬иә«еҚ з”ЁIPduдёӯдёҖдёӘBitдҪҚпјҢиҜҘBitдҪҚдёә1иЎЁзӨәеҜ№еә”Signal/SignalGroupжңүжӣҙж–°пјҢдёә0иЎЁзӨәеҜ№еә”Signal/SignalGroupжІЎжңүжӣҙж–°гҖӮ

    A Signal/SignalGroup determines whether to support the Update mechanism through the configuration parameter ComUpdateBitPosition. The Update Bit itself occupies one bit in the IPdu, where a value of 1 indicates the corresponding Signal/SignalGroup has been updated, and a value of 0 indicates it has not been updated.

  • еҜ№дәҺеҸ‘йҖҒдҝЎеҸ·пјҢеҪ“иҜ·жұӮSignal/SignalGroupеҸ‘йҖҒж—¶пјҢComжЁЎеқ—е°Ҷе…¶UpdateдҪҚзҪ®1пјҢиЎЁзӨәиҜҘдҝЎеҸ·жңүжӣҙж–°пјҢйҖҡиҝҮComTxIPduзҡ„й…ҚзҪ®еҸӮж•°ComTxIPduClearUpdateBitеҶіе®ҡд»Җд№Ҳж—¶еҖҷжё…йҷӨ(зҪ®0)иҜҘеҸ‘йҖҒIPduдёӯжүҖжңүзҡ„UpdateдҪҚгҖӮ

    For transmitted signals, when sending a Signal/SignalGroup is requested, the Com module sets its Update Bit to 1, indicating that the signal has been updated. The timing for clearing (setting to 0) all Update Bits in the transmitted IPdu is determined by the configuration parameter ComTxIPduClearUpdateBit of the ComTxIPdu.

  • еҜ№дәҺжҺҘж”¶дҝЎеҸ·пјҢеҸӘжңүеҪ“жЈҖжөӢеҲ°Signal/SignalGroupзҡ„UpdateдҪҚдёә1пјҢжүҚжү§иЎҢиҝӣдёҖжӯҘзҡ„жҺҘж”¶ж“ҚдҪңпјҢеҗҰеҲҷиҲҚејғиҜҘдҝЎеҸ·гҖӮ

    For received signals, further reception operations are performed only when the Update Bit of the Signal/SignalGroup is detected as 1; otherwise, the signal is discarded.

  • дёәдәҶеңЁеӨҡеҲҶеҢә(ж ё)д№Ӣй—ҙжҸҗдҫӣиҙҹиҪҪеҲҶй…ҚпјҢйҖҡдҝЎж Ҳзҡ„дёҚеҗҢйғЁеҲҶеҸҜд»Ҙиў«еҲҶй…ҚеҲ°дёҚеҗҢзҡ„еҲҶеҢәгҖӮдёҚеҗҢзҡ„зҪ‘з»ңзұ»еһӢеҰӮ FlexRayпјҢCAN е’Ң Ethernet зҡ„йғЁеҲҶеҸҜд»Ҙиў«еҲҶй…ҚеҲ°дёҚеҗҢзҡ„еҲҶеҢә(ж ё)гҖӮ

    To provide load distribution among multiple partitions (cores), different parts of the communication stack can be allocated to different partitions. Components of different network types such as FlexRay, CAN, and Ethernet can be assigned to separate partitions (cores).

  • дёәдәҶж”ҜжҢҒдёҠиҝ°зҒөжҙ»зҡ„еҲҶй…ҚпјҢеҮҸе°‘еҲҶеҢәй—ҙзҡ„йҖҡдҝЎ(д»ҺиҖҢжҪңеңЁзҡ„еҮҸе°‘и·ЁеҲҶеҢәеҗҢжӯҘеёҰжқҘзҡ„йҳ»еЎһ)пјҢCOM жЁЎеқ—еҸҜд»ҘжҢүз…§зҪ‘з»ңзұ»еһӢеҲӣе»әдёҚеҗҢзҡ„ MainFunction(жҜҸдёӘеҲҶеҢәиҮіе°‘дёҖдёӘ)гҖӮеңЁе…·дҪ“зҡ„ MainFunction дёӯд»…еӨ„зҗҶдёҺиҜҘзҪ‘з»ңзұ»еһӢзӣёе…ізҡ„ PDUпјҢжҺҘж”¶е’ҢеҸ‘йҖҒе°ҶдҝқжҢҒеңЁеҚ•дёӘзҪ‘з»ңиҢғеӣҙеҶ…(еҚіеңЁеҚ•дёӘеҲҶеҢәдёӯ)пјҢеӣ жӯӨдёҚйңҖиҰҒиҖғиҷ‘и·ЁеҲҶеҢәеҗҢжӯҘзҡ„й—®йўҳгҖӮе”ҜдёҖзҡ„дҫӢеӨ–жҳҜдҝЎеҸ·зҪ‘е…іпјҢеҪ“дҝЎеҸ·зҪ‘е…ізҡ„жәҗе’Ңзӣ®зҡ„PDUдёҚеңЁеҗҢдёҖдёӘеҲҶеҢәж—¶пјҢCOMжЁЎеқ—жҸҗдҫӣи·ЁеҲҶеҢәзҡ„ж•°жҚ®дёҖиҮҙжҖ§дҝқжҠӨгҖӮжҜҸдёӘ MainFunction йғҪжӢҘжңүзӢ¬з«Ӣзҡ„ Time BaseгҖӮ

    To support the above flexible allocation and reduce inter-partition communication (thereby potentially reducing blocking caused by cross-partition synchronization), the COM module can create different MainFunctions according to network types (at least one per partition). Each specific MainFunction only processes PDUs related to that network type, and reception and transmission are kept within a single network scope (i.e., within a single partition), thus eliminating the need to consider cross-partition synchronization. The only exception is the signal gateway: when the source and destination PDUs of the signal gateway are not in the same partition, the COM module provides cross-partition data consistency protection. Each MainFunction has an independent Time Base.

  • ComIPdu йҖҡиҝҮ ComIPduMainFunctionRef дёҺзү№е®ҡзҡ„ ComMainFunctionRx/ComMainFunctionTx е…іиҒ”гҖӮComMainFunctionRouteSignalsRef д»…еңЁдҝЎеҸ·зҪ‘е…ізӣ®ж ҮдҝЎеҸ·жүҖеңЁзҡ„ ComIPdu дёҠдҪҝз”ЁгҖӮComMainRxPartitionRef/ComMainTxPartitionRef/ComMainRouteSignalsPartitionRef еҲҶеҲ«иЎЁзӨәиҜҘ ComMainFunction е®һдҫӢиҝҗиЎҢзҡ„еҲҶеҢәгҖӮеҰӮжһңй…ҚзҪ®дәҶеҲҶеҢәдҝЎжҒҜпјҢеҲҷ ComMainFunction жүҖеңЁзҡ„еҲҶеҢәеҝ…йЎ»дёҺ ComIPdu е…іиҒ”зҡ„ Pdu(EcucPduDefaultPartitionRef)зҡ„еҲҶеҢәдёҖиҮҙгҖӮжӯӨеӨ„ејәи°ғпјҢеҰӮжһңеҸ‘йҖҒIPDU(IPDUжҲ–иҖ…IPDUеҢ…еҗ«зҡ„дҝЎеҸ·е’Ңз»„дҝЎеҸ·)з”ЁдәҺдҝЎеҸ·зҪ‘е…іпјҢеҲҷе…¶йңҖиҰҒеҗҢж—¶й…ҚзҪ®ComIPduMainFunctionRef е…іиҒ”еҲ°ComMainFunctionTx е’Ңй…ҚзҪ®ComMainFunctionRouteSignalsRefпјҢе№¶дё”еҲҶеҢәдёҖиҮҙгҖӮ

    ComIPdu is associated with specific ComMainFunctionRx/ComMainFunctionTx through ComIPduMainFunctionRef. ComMainFunctionRouteSignalsRef is only used on the ComIPdu where the target signal of the signal gateway is located. ComMainRxPartitionRef/ComMainTxPartitionRef/ComMainRouteSignalsPartitionRef respectively indicate the partition in which the ComMainFunction instance runs. If partition information is configured, the partition where ComMainFunction is located must be consistent with the partition of the Pdu (EcucPduDefaultPartitionRef) associated with ComIPdu. It is emphasized here that if a transmitting IPDU (IPDU or the signals and group signals contained in the IPDU) is used for the signal gateway, it needs to be configured with ComIPduMainFunctionRef associated to ComMainFunctionTx and ComMainFunctionRouteSignalsRef at the same time, and the partitions must be consistent.

  • ComжЁЎеқ—еҰӮжһңй…ҚзҪ®дәҶComIpduGroupпјҢеҲҷе…¶еҝ…йЎ»иў«иҮіе°‘дёҖдёӘIPDUе…іиҒ”пјҢдё”е…іиҒ”еңЁеҗҢдёҖдёӘComIpduGroupзҡ„IPDUзҡ„еҲҶеҢәеҝ…йЎ»дёҖиҮҙгҖӮ

  • IPDUзҡ„еҲҶеҢәдҝЎжҒҜжҳҜеңЁECUCжЁЎеқ—й…ҚзҪ®е®һзҺ°зҡ„пјҢComжЁЎеқ—й…ҚзҪ®е№¶ж ЎйӘҢжҲҗеҠҹеҗҺпјҢеңЁECUCжЁЎеқ—еҸій”®Synchronize ModuleйҖүжӢ©ComжЁЎеқ—иҝӣиЎҢеҗҢжӯҘпјҢеҸҜд»Ҙе°ҶComжЁЎеқ—зҡ„еҲҶеҢәдҝЎжҒҜеҗҢжӯҘеҲ°ECUCжЁЎеқ—зҡ„IPDUдёҠгҖӮ

    The partition information of IPDU is configured and implemented in the ECUC module. After the Com module is configured and verified successfully, right-click вҖңSynchronize ModuleвҖқ in the ECUC module and select the Com module to synchronize, so that the partition information of the Com module can be synchronized to the IPDU of the ECUC module.

  • д»Јз Ғз”ҹжҲҗеҷЁдјҡдёәжҜҸдёҖдёӘ ComMainFunction з”ҹжҲҗдёҖдёӘеҮҪж•°(еЈ°жҳҺеңЁSchM_Com.h)гҖӮжҜҸдёҖдёӘ ComMainFunction зҡ„е®һдҫӢе’ҢAPIжҺҘеҸЈеңЁйӣҶжҲҗж—¶йңҖиҰҒиў«ж”ҫеңЁжӯЈзЎ®зҡ„еҲҶеҢәдёӯиў«и°ғз”ЁгҖӮе»әи®®ејҖеҸ‘йҳ¶ж®өжү“ејҖComConfigurationUseDetй…ҚзҪ®пјҢеӯҳеңЁдёҚеҗҲзҗҶзҡ„еҲҶеҢәи°ғз”Ёж—¶пјҢеҸҜд»ҘжҠҘDetй”ҷиҜҜгҖӮ

Std_ReturnType Com_TriggerTransmit(PduIdType TxIpduId, PduInfoType *PduInfoPtr)

Within this API, the upper layer module (called module) shall check whether the available data fits into the buffer size reported by PduInfoPtr->SduLength. If it fits, it shall copy its data into the buffer provided by PduInfoPtr->SduDataPtr and update the length of the actual copied data in PduInfoPtr->SduLength. If not, it returns E_NOT_OK without changing PduInfoPtr.
